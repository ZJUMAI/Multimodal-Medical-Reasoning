# Contributing to Awesome Medical Reasoning in Multimodal Foundation Models

Thank you for your interest in contributing! This document provides guidelines for contributing to this repository.

## 🎯 Ways to Contribute

### 1. Adding Papers

When adding a new paper, please ensure you include:
- **Title**: Full paper title
- **Authors**: All authors (at least first author)
- **Venue**: Conference/Journal name
- **Year**: Publication year
- **Link**: arXiv or official paper link
- **Code**: GitHub repository link (if available)
- **Brief Description**: One-sentence summary (optional but recommended)

### 2. Format Guidelines

#### Paper Entry Format

```markdown
| [Paper Title](https://arxiv.org/abs/XXXX.XXXXX) | Venue | Year | Keywords | [🔗](https://github.com/username/repo) |
```

#### New Section Format

```markdown
### Section Title

Brief description of what this section covers.

| Paper | Venue | Year | Keywords | Code |
|-------|-------|------|----------|------|
| [Example](link) | Venue | Year | Tags | [🔗](link) |
```

## 📋 Pull Request Process

1. **Fork the Repository**
   ```bash
   git clone https://github.com/YOUR_USERNAME/YOUR_REPO.git
   cd YOUR_REPO
   ```

2. **Create a New Branch**
   ```bash
   git checkout -b add-paper-name
   ```

3. **Make Your Changes**
   - Add your content following the format guidelines
   - Ensure no duplicate entries exist
   - Check that all links work

4. **Commit Your Changes**
   ```bash
   git add .
   git commit -m "Add: [Paper Name] in [Section]"
   ```

5. **Push and Create PR**
   ```bash
   git push origin add-paper-name
   ```
   Then create a Pull Request on GitHub

## ✅ Checklist Before Submitting

- [ ] Paper is placed in the correct section
- [ ] All links are working
- [ ] No duplicate entries
- [ ] Follows the existing format
- [ ] Includes all required information
- [ ] Commit message is clear and descriptive

## 🏷️ Commit Message Guidelines

- `Add: [Paper Name]` - Adding new paper
- `Update: [Section Name]` - Updating existing content
- `Fix: [Issue Description]` - Fixing errors
- `Docs: [Description]` - Documentation updates
- `Refactor: [Description]` - Restructuring content

## 🔍 Quality Standards

### Required Information
- ✅ Valid paper link (arXiv, official venue, or preprint)
- ✅ Correct venue and year
- ✅ Working code link (if available)

### Preferred Information
- ⭐ Brief description or key contribution
- ⭐ Tags/keywords for categorization
- ⭐ Performance metrics (if applicable)

## 📝 Suggesting New Categories

If you believe a new category should be added:

1. Open an issue titled: "New Category: [Category Name]"
2. Provide:
   - Rationale for the new category
   - At least 3-5 papers that would fit
   - Suggested placement in the existing structure

## 🐛 Reporting Issues

When reporting issues, please include:
- Clear description of the problem
- Section/line where the issue occurs
- Suggested fix (if applicable)

### Issue Labels
- `bug` - Something isn't working
- `enhancement` - New feature or request
- `documentation` - Improvements to docs
- `good first issue` - Good for newcomers
- `paper-addition` - Adding new paper
- `help wanted` - Extra attention needed

## 🤝 Code of Conduct

### Our Standards

- ✅ Be respectful and inclusive
- ✅ Provide constructive feedback
- ✅ Focus on what is best for the community
- ✅ Show empathy towards other contributors

### Unacceptable Behavior

- ❌ Harassment or discriminatory language
- ❌ Trolling or insulting comments
- ❌ Publishing others' private information
- ❌ Other unprofessional conduct
